# Velo 重构报告

## 1. 命名与坐标方案

本次没有采用 `io.velo` 这类脱离仓库归属的坐标，而是统一为：

- `groupId`: `io.github.luminion`
- Java 包名: `io.github.luminion.velo`
- 配置前缀: `velo.*`

这样做的原因：

- 保留 Maven Central / GitHub 归属一致性，避免后续发布时坐标和账号脱节。
- 对使用者来说，`velo` 是产品名，`io.github.luminion` 是发布者名，职责更清晰。

父工程和模块命名同步调整为更接近 Spring 官方语义的形式：

- 父工程: `velo-spring-boot-project`
- 版本依赖模块:
  - `velo-spring-boot2-dependencies`
  - `velo-spring-boot3-dependencies`
  - `velo-spring-boot4-dependencies`
- 对外 starter / autoconfigure 模块:
  - `velo-spring-boot2-starter` / `velo-spring-boot2-autoconfigure`
  - `velo-spring-boot3-starter` / `velo-spring-boot3-autoconfigure`
  - `velo-spring-boot4-starter` / `velo-spring-boot4-autoconfigure`

## 2. 结构改造

已完成 `lumos -> velo` 的全链路迁移：

- 模块目录名、`pom.xml` 中的 `artifactId` / `name` / `parent` / `dependencyManagement`
- Java 类名 `Lumos* -> Velo*`
- 包名 `io.github.lumos.starter -> io.github.luminion.velo`
- 自动装配注册文件
  - Boot 2: `META-INF/spring.factories`
  - Boot 3/4: `META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports`
- `README.md` 配置示例与依赖示例

## 3. 公共模块下沉

按“无 `javax/jakarta` 差异就放 common”的原则，已下沉到 `velo-autoconfigure-common`：

- `VeloCoreAutoConfiguration`
- `VeloCacheAutoConfiguration`
- `VeloRedisAutoConfiguration`
- `VeloDateTimeFormatAutoConfiguration`

对应测试也一并迁入 common：

- `VeloCacheAutoConfigurationTests`
- `VeloRedisAutoConfigurationTests`
- `VeloDateTimeFormatAutoConfigurationTests`

这样做后：

- `velo-autoconfigure-javax` 和 `velo-autoconfigure-jakarta` 只保留真正和 Servlet / `javax` / `jakarta` 绑定的能力。
- 公共模块边界更接近 Spring 官方的“公共自动装配 + 差异适配层”结构。

## 4. 开关与装配优化

### 4.1 自动装配开关

本轮保留并统一了现有的 `@ConditionalOnProperty` 开关体系，全部迁移到 `velo.*` 前缀。

重点保留/强化的开关包括：

- `velo.core.*`
- `velo.cache.*`
- `velo.redis.*`
- `velo.date-time-format.*`
- `velo.jackson.*`
- `velo.idempotent.*`
- `velo.rate-limit.*`
- `velo.lock.*`
- `velo.web.*`
- `velo.web.xss.*`
- `velo.log.*`

### 4.2 日志装配改进

日志切面原来依赖 `Slf4JLogWriter` bean 存在才会装配，导致：

- 一旦用户关闭默认 `Slf4J` writer
- 即使自己提供了 `InvokeArgsWriter` / `InvokeResultWriter` / `ErrorLogWriter` / `SlowLogWriter`
- 切面仍然不会生效

现在已经改成按接口类型判断：

- `ArgsLogAspect` 依赖 `InvokeArgsWriter`
- `ResultLogAspect` 依赖 `InvokeResultWriter`
- `ErrorLogAspect` 依赖 `ErrorLogWriter`
- `SlowLogAspect` 依赖 `SlowLogWriter`

这意味着自定义 writer 可以真正替代默认实现。

### 4.3 Redis Bean 选择改进

`Idempotent` / `RateLimit` 的 Redis 后端原来按 `RedisTemplate` 类型注入，存在多个 `RedisTemplate` 时可能出现歧义。

现在已改为：

- `@ConditionalOnBean(name = "redisTemplate")`
- `@Qualifier("redisTemplate")`

这样即使项目里同时存在 `redisTemplate` 和 `stringObjectRedisTemplate`，也能稳定选择预期的主 Redis Bean。

## 5. 优先级策略

对多后端能力，当前显式固定为：

- `Redisson > Redis > Caffeine > JDK`

覆盖范围：

- `Idempotent`
- `RateLimit`
- `Lock`

实现方式：

- `@AutoConfiguration(after = ...)`
- `@ConditionalOnMissingBean(...)`
- 后端细粒度开关 `velo.*.backends.*`

并补了测试，保证以下行为成立：

- Redisson 和 Redis 同时存在时，优先 Redisson
- 关闭 Redisson 后端时，自动回退到 Redis
- 没有外部后端时，再回退到 Caffeine / JDK

## 6. 依赖裁剪

在 `velo-autoconfigure-javax` / `velo-autoconfigure-jakarta` 中移除了已经下沉后不再需要的依赖：

- `spring-boot-starter-validation`
- `spring-boot-starter-data-redis`
- `spring-boot-starter-cache`

保留的依赖主要对应剩余职责：

- `spring-boot-starter-web`
- `spring-boot-starter-mail`
- `mybatis-plus-*`
- `jsoup`

## 7. 主动保留的差异点

以下能力我没有继续下沉到 common，是有意保留：

- `MyBatis-Plus` 自动装配

原因：

- Boot 2 / 3 / 4 对应的 starter 坐标分别是
  - `mybatis-plus-boot-starter`
  - `mybatis-plus-spring-boot3-starter`
  - `mybatis-plus-spring-boot4-starter`
- 虽然类名基本一致，但依赖坐标本身已经带有 Boot 版本语义
- 继续放在 `javax/jakarta` 层更稳妥，避免 common 模块编译依赖被 Boot 版本绑定

```bash
mvn -T 1C -pl velo-spring-boot2-starter,velo-spring-boot3-starter,velo-spring-boot4-starter -am -DskipTests compile
```

结果：

- 自动装配测试通过
- Jackson 测试通过
- common 模块测试通过
- Boot 2/3/4 自动装配清单测试通过
- starter 模块编译通过
